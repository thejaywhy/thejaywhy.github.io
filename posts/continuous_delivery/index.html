Book Report: Continuous Delivery


Continuous Delivery: Reliable Software Releases Through Build, Tes, and Deployment Automation by Jez Humble and David Farley.

This book is not a “how to” on building a Continuous Integration or Continuous Deployment Pipeline. Instead it gives details on why you should invest in the CD pipeline, and it’s principles. But, the book does recommend the following order for implementing a pipeline (p. 133):

  1. Model your value stream and createa walking skelton
  2. Automate teh build and deployent process
  3. Automate uit tests and code analysis
  4. Automate acceptance tests
  5. Automate releases.

Version controlled deployment scripts and configuration (see chef). Use the same deployment strategy for EVERY environment: dev, test, integration, staging, production. Exposes problem with deployment strategy BEFORE you get to production.

Acceptance Tests should be written by developers. A seperate QA team will lead to developers ignoring Acceptance Test failures because they don’t “own” them. “The Wwhole team owns the acceptance tests… if the acceptance tests fail, the whole team dhoul stop and fixthem immediately”.

Acceptance Tests -> Behavior Driven Testing/Development.

-> Acceptance Criteria: Given … When … Then … -> Test Implemntaion Layer DSL No reference to UI -> Application Driver Layer UI specific layer (or API)

USE PRODUCTION APIs - don’t build in a back door to make testing easier. That’s dangerous and you’ll spend extra time building and maintaining something that doesn’t give immediate business value.
